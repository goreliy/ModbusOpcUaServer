# План разработки: OPC UA сервер с опросом Modbus и формулами (Rust)

**Продукт — OPC UA сервер**, источником данных которого служит встроенный опросчик
Modbus RTU / Modbus TCP / Modbus RTU-over-TCP (любые функции), с движком тегов и
формулами преобразования, логированием, десктопным конфигуратором (egui), службой для
Windows и Linux (systemd, включая Astra Linux) и готовыми инсталляторами.
MQTT-публикация — вторичная опция, не ядро продукта.

> Зафиксированные решения по итогам обсуждения — см. §0 и §11. Этот документ обновляется
> по мере уточнения требований; версия ниже учитывает уже принятые решения.

## 0. Допущения и терминология

- **"OPC UA 2.0"** — как отдельного номера спецификации официально нет. Есть OPC UA 1.04/1.05
  (последняя добавила PubSub, MQTT-транспорт, JSON-кодирование). **Решено: нужны классический
  Client/Server (opc.tcp) и push-рассылка данных наружу.** Проверка крейтов показала, что
  готовой безопасной Rust-реализации спека-совместимого OPC UA PubSub нет (см. §2, §11) —
  выбран прагматичный путь: push-рассылка тегов реализуется через **MQTT/JSON**, а не через
  формальный OPC UA PubSub (UADP/PublishedDataSet). Функционально задача "рассылка изменений
  многим подписчикам" решена; спек-совместимость с чужими OPC UA PubSub стеками — не гарантирована.
- "Дэги" = теги (переменные адресного пространства OPC UA, источник значения — регистр/бит Modbus).
- Целевые ОС: Windows 10/11 + Windows Server (служба) и **любой Linux с systemd**
  (Astra Linux — один из целевых, но не единственный; собираем максимально совместимо,
  без специфичных для Astra допущений, кроме упаковки .deb).

## 1. Архитектура (workspace, набор крейтов)

```
opc-modbus-gateway/                 (cargo workspace)
├─ crates/
│  ├─ mb-proto/         протокол Modbus: кодеки RTU/TCP/RTU-over-TCP, любые функции, CRC
│  ├─ mb-poller/         движок опроса: шардирование по каналам, группы, периоды, retry/timeout, кэш значений
│  ├─ tags-core/         модель тегов, типы данных, byte/word order, formula-engine, короткий буфер истории
│  ├─ config/            схема конфигурации (serde, JSON), валидация (jsonschema), hot-reload
│  ├─ opcua-gateway/      OPC UA сервер (async-opcua): адресное пространство, subscriptions, запись
│  ├─ mqtt-publisher/     push-рассылка тегов по MQTT/JSON (замена спек-PubSub), общий trait с opcua-gateway
│  ├─ diag-log/           tracing-логирование, ротация, журнал трафика Modbus, счётчики/диагностика
│  ├─ svc-host/           обёртка службы: Windows Service / systemd unit, graceful shutdown
│  ├─ gui/                десктопный конфигуратор (Windows+Linux), редактирует JSON-конфиг
│  └─ app/                main.rs служебного процесса + CLI (валидация конфига, dry-run опроса)
├─ installers/
│  ├─ windows/            WiX (cargo-wix) → .msi (служба + GUI-конфигуратор)
│  └─ linux/              cargo-deb (.deb) + systemd unit, опционально .rpm/AppImage
├─ tests/                 интеграционные тесты (симулятор Modbus-слейва, тестовый OPC UA/PubSub клиент)
└─ docs/
```

Поток данных:

```
[RTU-устройство]──serial──┐
[TCP-устройство]──TCP─────┼──> mb-poller (шардированный планировщик, per-канал/группа) ──> tags-core
[RTU-over-TCP]────TCP─────┘         (конкурентный кэш: значение + quality + timestamp)
                                              │
                                              ▼
                                     formula-engine (масштабирование/выражения) + short history buffer
                                              │
                                              ▼
                        opcua-gateway (адресное пространство, подписки, запись назад)  +  mqtt-publisher (push JSON)
                                    │                              │
                                    ▼                              ▼
                          OPC UA клиенты (opc.tcp)        Подписчики MQTT (JSON payload)
```

Отдельно: **gui** — десктопное приложение конфигурирования, работает с тем же JSON-файлом,
что и служебный процесс; может запускаться на том же узле или удалённо редактировать конфиг
для последующей заливки на объект.

## 2. Технологический стек

| Задача | Крейт/технология | Комментарий |
|---|---|---|
| Асинхронный runtime | `tokio` | единый для поллера, службы, GUI-фона |
| Modbus протокол | `tokio-modbus` (+ доработка) | RTU/TCP из коробки; произвольные функции — через `Request::Custom`/сырой PDU |
| Serial-порт (RTU) | `tokio-serial` | Windows (COMx) и Linux (/dev/ttyUSBx, /dev/ttyS0) |
| RTU-over-TCP | собственный транспорт | RTU-кодек (адрес+PDU+CRC16) поверх `TcpStream`, без MBAP |
| **OPC UA сервер (Client/Server)** | **`async-opcua`** (FreeOpcUa, чистый Rust, форк locka99/opcua) | Client/Server, opc.tcp, подписки, SecurityPolicy (в т.ч. Basic256Sha256) — без C-тулчейна/cmake, кросс-компиляция как у обычного Rust-крейта. PubSub в нём не реализован — закрываем отдельно (см. ниже) |
| Push-рассылка тегов ("PubSub"-замена) | `rumqttc` (чистый Rust MQTT-клиент) + JSON (`serde_json`) | Публикация изменений тегов в MQTT-топики как JSON — не спек-совместимый OPC UA PubSub, но решает практическую задачу 1:M рассылки. | Формулы/выражения | `evalexpr` или `fasteval` | линейное масштабирование A·x+B "из коробки" + произвольные выражения |
| Конфигурация | `serde` + **JSON** (`serde_json`) | версия схемы, `jsonschema`/`schemars` для валидации и для автогенерации форм GUI |
| Логирование | `tracing` + `tracing-subscriber` + `tracing-appender` | ротация файлов, уровень по модулям, отдельный лог трафика Modbus |
| Служба Windows | `windows-service` | автозапуск, Event Log |
| Служба Linux | systemd unit (`Restart=on-failure`) | универсально для любого systemd-дистрибутива, включая Astra |
| Короткий буфер истории | `sled` (embedded KV) | retentive-теги, последние N значений/аварий, без полноценных трендов |
| Десктоп-GUI | `egui`/`eframe` — **решено, Tauri исключён** | Чистый Rust, один бинарник, без WebView-зависимости (на Linux не нужен webkit2gtk, на Windows — WebView2); рендер через wgpu/glow работает на обеих ОС |
| Конкурентные структуры для масштаба | `dashmap`, `tokio::sync::watch/broadcast` | кэш тегов и рассылка изменений при большом числе устройств/тегов без глобальных блокировок |
| CLI | `clap` | валидация конфига, dry-run опроса, экспорт адресного пространства |
| Инсталлятор Windows | `cargo-wix` → WiX Toolset | .msi: служба + GUI-конфигуратор, ярлыки, правило Firewall (порт 4840 + PubSub UDP/порт MQTT) |

| CI | GitHub Actions/GitLab CI, `x86_64-pc-windows-msvc` и `x86_64-unknown-linux-gnu` (+ musl опция) | сборка + тесты + сборка инсталляторов на каждый тег; чистый Rust-стек — без cmake/C-тулчейна |

## 3. Опросчик Modbus (mb-proto + mb-poller)

### 3.1 Транспорты
- **Modbus RTU** — COM/tty, настройки: baud, data bits, parity, stop bits, межкадровая пауза,
  таймаут ответа, задержка между запросами (для медленных RS-485 конвертеров), режим RS-485
  (RTS toggle) при необходимости через драйвер ОС.
- **Modbus TCP** — host:port, unit id, keepalive, таймаут, реконнект с backoff.
- **Modbus RTU-over-TCP** ("сериал-туннель") — тот же RTU-кадр (адрес+PDU+CRC16), но передаётся
  как есть по TCP-соединению (без MBAP), как у типовых серийных серверов (Moxa NPort и т.п.).

### 3.2 Функции Modbus (полный набор + расширяемость)
- Стандартные: 01 Read Coils, 02 Read Discrete Inputs, 03 Read Holding Registers,
  04 Read Input Registers, 05 Write Single Coil, 06 Write Single Register,
  15 Write Multiple Coils, 16 Write Multiple Registers, 22 Mask Write Register,
  23 Read/Write Multiple Registers.
- Диагностика/сервис: 07 Read Exception Status, 08 Diagnostics, 11 Get Comm Event Counter,
  17 Report Slave ID, 43 Encapsulated Interface (device identification).
- **Произвольная/вендорская функция** — универсальный "raw PDU" режим (код функции + байты
  запроса/ответа задаются в конфиге), чтобы закрыть нестандартные устройства без правки кода.

### 3.3 Планировщик опроса — рассчитан на крупный масштаб (>100 устройств, >5000 тегов)
- Устройство → один или несколько **каналов связи** (канал — физическое соединение;
  на одной RS-485 линии может сидеть много слейвов, шарящих канал).
- **Шардирование**: каждый физический канал (COM-порт/TCP-сокет) обслуживается отдельной
  async-задачей/воркером; независимые каналы опрашиваются параллельно, без взаимной блокировки.
- **Группы опроса** с независимым периодом ("быстрые" 200 мс, "медленные" 5 с, on-demand
  при активной OPC UA подписке/PubSub-публикации).
- Для одного физического RTU-канала — строго последовательный опрос (шина полудуплексная),
  очередь запросов с приоритетами; TCP-устройства опрашиваются конкурентно с ограничением
  максимального числа одновременных соединений/запросов (защита от перегрузки сети/CPU).
- Retry/timeout policy на группу и на устройство, экспоненциальный backoff при потере связи,
  watchdog "устройство offline" → quality Bad для всех его тегов.
- Coalescing — слияние соседних регистров в один запрос по границе адресов и максимального
  размера ответа, критично при тысячах тегов, чтобы не упереться в лимит запросов/сек.
- Конкурентный кэш тегов (`dashmap` + версии/timestamps) — чтение из OPC UA/PubSub-слоя без
  конкуренции с потоками опроса.
- Метрики на канал/устройство: успешные/ошибочные опросы, CRC-ошибки, таймауты, round-trip,
  загрузка шины (%), которые нужно замерить нагрузочным тестом ещё на этапе MVP (см. §10) —
  на крупном объекте важно заранее понять предел по числу тегов/период на один RTU-канал.

## 4. Модель тегов и формулы (tags-core)

- **Тип данных источника**: bit, uint16/int16, uint32/int32, uint64/int64, float32, float64,
  BCD, ASCII-строка из диапазона регистров, bitfield (маска/битовые поля из одного регистра).
- **Порядок байт/слов**: big/little endian на уровне байт и на уровне слов ("swap words" у
  ПЧ и счётчиков электроэнергии), настраивается на тег или на устройство.
- **Формулы**:
  - линейное масштабирование `value = raw * scale + offset` с инженерными единицами и
    диапазоном (сигнализация выхода за диапазон);
  - произвольное выражение (`evalexpr`) с переменной `raw` и доступом к другим тегам
    (сложные пересчёты, например суммирование фаз);
  - обратное преобразование для **записи** (OPC UA write → Modbus write), либо явная
    отдельная формула записи, если пересчёт не симметричен;
  - deadband/фильтр перед публикацией изменения (снижение потока событий в OPC UA/PubSub).
- **Качество и метка времени**: Good/Uncertain/Bad, timestamp опроса, устаревание значения
  при потере связи.
- **Короткий буфер истории** (`sled`): последние N значений/переходов на тег, retentive-теги
  (сохраняются между перезапусками) — без полноценных трендов, тренды остаются за
  внешней SCADA/MES.

## 5. OPC UA сервер (opcua-gateway, на `async-opcua`) + push-рассылка (mqtt-publisher)

- Автогенерация адресного пространства из списка тегов (папки по устройствам/группам),
  тип узла — Variable с DataType по типу тега, атрибуты EngineeringUnits/Range.
- **Subscriptions/MonitoredItems (классический режим)** — публикация изменений из кэша
  поллера (push по deadband), периодическая выборка, если клиент просит чаще периода опроса
  (сервер отдаёт последнее известное значение — физически чаще не опросить).
- **Push-рассылка по MQTT** — выбранные наборы тегов публикуются как JSON-сообщения в
  MQTT-топики (структура топика и период публикации настраиваются в конфиге, независимо
  от периода опроса Modbus). Реализовано через общий trait `TagPublisher`, чтобы при
  необходимости заменить/дополнить реализацию спек-совместимым OPC UA PubSub без
  переписывания вызывающего кода в mb-poller/tags-core.
- **Запись (Write service)** — валидация диапазона/типа → очередь write-запроса Modbus
  с приоритетом выше обычного цикла чтения.
- **On-demand опрос** — тег с активной подпиской/PubSub-публикацией подключается к группе
  опроса динамически (не опрашивать то, что никому не нужно).
- **Безопасность — с релиза 1.0**: SecurityPolicy Basic256Sha256, режимы Sign/SignAndEncrypt,
  аутентификация Username/Password (плюс возможность добавить сертификаты клиентов позже),
  самоподписанный сертификат сервера при первом запуске, хранение учётных данных зашифрованно
  на диске (не plain-text в конфиге).
- **Диагностические узлы**: состояние каждого канала/устройства (Online/Offline), счётчики
  ошибок, время последнего успешного опроса, загрузка шины — обычные OPC UA переменные.
- Namespace URI и NodeId-схема фиксируются в конфиге (стабильность адресного пространства
  между перезапусками).

## 6. Логирование и диагностика (diag-log)

- Прикладной лог (`tracing`): запуск/останов, ошибки конфигурации, ошибки OPC UA
  сессий/PubSub, ротация по размеру/дате, уровень настраивается на модуль.
- **Трафик Modbus** — отдельный включаемый по устройству/каналу лог "сырых" кадров
  (hex-дамп запрос/ответ + таймстамп) для диагностики на объекте без Wireshark.
- Интеграция с журналами ОС: Windows Event Log, Linux syslog/journald.
- Метрики опроса, OPC UA сессий и PubSub-публикаций — опционально экспорт в Prometheus
  (`/metrics`), важно на крупном объекте для мониторинга нагрузки шин и очередей.

## 7. Кроссплатформенность и служба (svc-host)

- Единый бинарник, конфиг и бизнес-логика общие; платформенные различия только в тонком
  слое: serial-порт (`tokio-serial` абстрагирует), служба (Windows Service API vs systemd),
  пути конфигурации/логов (`%ProgramData%` vs `/etc` + `/var/log`).
- **Windows**: регистрация как Windows Service (`windows-service`), автозапуск,
  восстановление после сбоя через Recovery Actions, логи в Event Log + файл.
- **Linux (любой с systemd, включая Astra)**: systemd unit (`Restart=on-failure`,
  `WantedBy=multi-user.target`), конфиг в `/etc/opc-modbus-gateway/`, логи в
  `/var/log/opc-modbus-gateway/` + journald. Собираем под glibc как основной таргет;
  musl — опциональная сборка для максимальной переносимости на нестандартные/старые системы.
- Права на серийные порты на Linux (группа `dialout`) — инсталлятор добавляет сервисного
  пользователя в группу автоматически.
- **GUI-конфигуратор** ставится тем же инсталлятором как отдельное desktop-приложение
  (не служба), может редактировать конфиг локально или удалённо (файл с диска/сетевого пути).

## 8. Инсталляторы

- **Windows**: `cargo-wix` → единый `.msi` — копирует бинарник службы и GUI-конфигуратор,
  дефолтный JSON-конфиг, регистрирует и запускает службу, ярлыки в "Пуск"/"Программы и
  компоненты", правило Firewall для opc.tcp (порт 4840) и PubSub (UDP-порт/MQTT-порт).
- **Linux**: `.deb` через `cargo-deb` — systemd unit, `postinst` создаёт пользователя/группу,
  каталоги `/etc`, `/var/log`, `/var/lib`, ставит GUI (desktop-файл + иконка), включает и
  стартует службу; опционально `.rpm`/AppImage для не-Debian систем.
- Оба инсталлятора кладут пример конфигурации и документацию рядом, не перезаписывая
  существующий конфиг при обновлении версии.

## 9. Тестирование

- Юнит-тесты кодеков Modbus (CRC, кадрирование RTU, MBAP TCP) на эталонных байтовых
  последовательностях.
- Встроенный **Modbus-слейв симулятор** (для CI и для стенда наладчика) — эмулирует
  RTU/TCP устройство с настраиваемыми регистрами, задержками, обрывами связи, в том числе
  массово (сотни виртуальных устройств) для нагрузочных тестов.
- Интеграционные тесты: опрос симулятора → кэш тегов → проверка значений через тестовый
  OPC UA клиент (классический) и тестовый PubSub-подписчик (UDP/MQTT).
- **Нагрузочный тест — приоритетно, начиная с MVP** (масштаб объекта заявлен как крупный:
  >100 устройств, >5000 тегов): предельное число тегов/устройств на канал при заданных
  периодах опроса, деградация при насыщении RTU-шины, поведение при массовых обрывах связи.
- Кросс-платформенный прогон CI на Windows-раннере и Linux-раннере (generic systemd-образ,
  дополнительно — прогон на реальном образе Astra Linux перед релизом).

## 11. Зафиксированные решения (по итогам обсуждения)

| Вопрос | Решение | Влияние на план |
|---|---|---|
| PubSub/MQTT | Нужен с релиза 1.0; спек-совместимого безопасного Rust-PubSub на рынке нет | OPC UA — чистый Rust `async-opcua`; push-рассылка — MQTT/JSON через `rumqttc`, за trait-абстракцией `TagPublisher` |
| Масштаб объекта | Крупный (>100 устройств, >5000 тегов) | Шардированный планировщик, coalescing, нагрузочные тесты с фазы 1 |
| Историзация | Короткая встроенная буферизация | `sled` в tags-core, без полноценных трендов (тренды — задача SCADA/MES) |
| Безопасность OPC UA | Basic256Sha256 + Username/Password | Реализуется в фазе 5, не откладывается на "потом" |
| Конфигуратор | Desktop GUI + JSON-файл конфигурации | Отдельный крейт `gui`, JSON вместо YAML, схема через `schemars`/`jsonschema` |
| GUI-тулкит | **`egui`/`eframe`; Tauri исключён** (решение заказчика) | Чистый Rust, один бинарник, никаких WebView-зависимостей на целевых машинах |
| Целевой Linux | Любой Linux с systemd (не только Astra) | Универсальная сборка glibc + опция musl, .deb как основной формат, без Astra-специфики |
| Реестр отечественного ПО | Не требуется | Свобода выбора крейтов/лицензий, включая C-зависимость open62541 |

## 12. Ключевые риски

- **`async-opcua` — активный форк, но моложе классического `opcua` (locka99)** — на фазе 0
  нужно подтвердить, что нужный набор SecurityPolicy (Basic256Sha256) и Username/Password
  аутентификация реализованы и стабильны в используемой версии, до того как на них
  завязывается фаза 5 (безопасность).
- **MQTT/JSON — не спек-совместимый OPC UA PubSub.** Если у заказчика/интегратора появится
  жёсткое требование интеропа именно с OPC UA PubSub подписчиками (не MQTT/JSON вообще, а
  конкретно UADP/PublishedDataSet), понадобится unsafe-слой на `open62541-sys` — заложено
  как альтернативная реализация trait `TagPublisher`, но не в объёме релиза 1.0.
- Нужен MQTT-брокер (Mosquitto/EMQX и т.п.) как внешняя инфраструктура для push-рассылки —
  это не входит в поставку продукта, нужно явно документировать для заказчика.
- **Крупный масштаб** — реальный предел пропускной способности RTU-шины физический
  (baud rate, число устройств на линии); никакая архитектура ПО это не обойдёт — нужно
  зафиксировать с заказчиком реалистичные периоды опроса для конкретных линий связи.
- **GUI-тулкит — решено: `egui`/`eframe`, Tauri исключён** (решение заказчика). Чистый Rust
  без WebView-зависимости: на минимальных Linux-установках нет webkit2gtk, и это не проблема.
  Остаточный риск egui — сложные таблицы/формы придётся собирать из примитивов (нет готовых
  "гридов" уровня web), закладываем это в оценку фазы 7.
- `tokio-modbus` закрывает стандартные функции "из коробки"; произвольные функции — через
  `Custom(u8, Bytes)`, декодирование ответа для нестандартных функций пишется самостоятельно.
- RTU-over-TCP не стандартизован единообразно у всех вендоров серийных серверов — нужно
  зафиксировать конкретные модели шлюзов, с которыми обязательна совместимость.
